# Жизненный цикл импорта

Этот документ отвечает на главный вопрос проекта: когда данные существуют только
локально, когда сервер их временно обрабатывает и в какой момент изменяется
PostgreSQL.

## Короткий ответ

После выбора и проверки файла он **не сохраняется в PostgreSQL**. Бинарный
CSV/XLS/XLSX вообще не хранится в БД. До финального подтверждения в браузере
находится локальный черновик с разобранными строками. Запись в PostgreSQL
происходит один раз: по кнопке «Импортировать контакты» через
`POST /api/imports/commit`.

## Состояния импорта

```text
Файл на компьютере
        |
        | POST /api/upload: временный multipart-запрос
        v
Проверенный локальный черновик (IndexedDB, 24 часа)
        |
        | редактирование + POST /api/imports/validate
        v
Локально проверенный черновик
        |
        | POST /api/imports/preview: только SELECT
        v
Предпросмотр конфликтов + локальные решения
        |
        | POST /api/imports/commit: одна транзакция
        v
Подтверждённые uploaded_files + file_rows + contacts + contact_sources
```

## Этап 1. Выбор и первичная проверка

Пользователь выбирает CSV, XLS или XLSX и нажимает «Проверить файл».
`handlers.UploadHandler.Upload` ограничивает размер запроса, получает файл из
multipart-формы и вызывает `services.ParseByFilenameWithOptions`.

Сервис определяет реальный формат по сигнатуре, разбирает CSV/Excel, находит
заголовки, очищает ячейки, нормализует телефон, email и скидку, формирует ошибки
строк и статистику. `services.NewImportValidation` присваивает UUID `importId` и
возвращает все строки как `models.ImportDraft`.

`UploadHandler` не имеет поля `store`, поэтому физически не может вызвать
PostgreSQL. Временные multipart-файлы удаляются после обработки запроса.

## Этап 2. Локальная работа

`frontend/js/draft-store.js` сохраняет ответ в IndexedDB `task1-local`. Черновик
живёт 24 часа, после чего удаляется при следующем открытии приложения.

Пользователь может:

- просматривать и редактировать все строки постранично;
- искать подстроку по всему черновику с подсветкой совпадений;
- перейти к строкам с ошибками и исправить значения;
- отменить импорт из сводки или непосредственно из финального этапа;
- выбрать другой файл.

Обе кнопки «Отменить импорт» вызывают один сценарий: после подтверждения удаляют
черновик из IndexedDB и очищают интерфейс. Серверный запрос не выполняется, а
PostgreSQL остаётся без изменений.

Все эти действия, включая предпросмотр конфликтов и финальное подтверждение,
находятся на первой вкладке «Файлы». Переход в отдельный экран контактов для
завершения импорта не требуется.

Каждая правка сразу меняет единый объект черновика и с задержкой 300 мс
сохраняется в IndexedDB с признаком `dirty`. Dirty-черновик нельзя сравнивать с
БД или импортировать, пока пользователь не нажмёт «Проверить изменения».

## Этап 3. Повторная проверка

`POST /api/imports/validate` принимает только черновик. Сервер не доверяет
статистике браузера: `services.ValidateImportDraft` заново проверяет UUID,
метаданные, заголовки, номера и значения строк, а затем пересчитывает ошибки и
статистику. Хранилище в этом обработчике не вызывается.

Обоснование этапа: пользователь может исправить телефон, email, ФИО или скидку,
но клиентская правка должна пройти те же правила, что исходный файл.

## Этап 4. Предпросмотр PostgreSQL

`POST /api/imports/preview` повторно валидирует черновик и вызывает
`storage.FindContactsByPhones`. SQL содержит только `SELECT ... WHERE phone =
ANY($1)`; `INSERT`, `UPDATE`, транзакции записи и блокировки здесь отсутствуют.

Каждая строка классифицируется:

- `new` — телефона ещё нет;
- `matched` — телефон и все фиксированные данные совпадают;
- `conflict` — телефон тот же, но ФИО, email или скидка отличаются;
- `skipped` — строка не даёт контакта.

Для конфликта сервер возвращает `updated_at` как `version`. Действия `skip`,
`replace` и `merge` пользователь выбирает локально. До финальной кнопки они не
отправляются в PostgreSQL.

Список конфликтов отображается по 25 записей на страницу. При переключении
страниц JavaScript удаляет предыдущие карточки из DOM и создаёт следующую порцию,
но уже выбранные решения остаются в `appState.decisions`. Поэтому пользователь
может пройти несколько страниц, вернуться назад и не потерять выбор. Новый запрос
к backend при перелистывании не нужен: полный результат уже вернул безопасный
предпросмотр, который выполняет только `SELECT`.

## Этап 5. Финальный commit

`POST /api/imports/commit` снова валидирует черновик, заново читает актуальные
контакты и проверяет решения. Затем `storage.PostgresStorage.CommitImport`
открывает транзакцию уровня `SERIALIZABLE`.

Внутри транзакции последовательно выполняются:

1. `INSERT uploaded_files` с метаданными подтверждённого импорта.
2. `COPY file_rows` со всеми нормализованными строками.
3. Advisory lock для каждого телефона в одинаковом сортированном порядке.
4. `SELECT ... FOR UPDATE` актуального контакта.
5. `INSERT contacts`, совпадение без изменения или выбранный `UPDATE contacts`.
6. `INSERT ... ON CONFLICT` связи `contact_sources`.
7. `COMMIT`.

Если importId уже использован, версия контакта устарела, появился параллельный
контакт или любой SQL завершился ошибкой, выполняется rollback. Частично
сохранённого файла или наполовину обновлённого справочника не остаётся.

## Обоснование пользовательских функций

| Функция | Зачем нужна | Изменяет PostgreSQL |
| --- | --- | --- |
| Проверить файл | Показать структуру и ошибки до принятия решения | Нет |
| Редактировать строки | Исправить данные без побочных эффектов | Нет |
| Локальный поиск | Быстро найти значение во всём черновике | Нет |
| Проверить изменения | Повторить серверные правила после ручной правки | Нет |
| Отменить импорт | Безопасно передумать и удалить IndexedDB-черновик | Нет |
| Проверить конфликты | Увидеть последствия на актуальном справочнике | Только SELECT |
| Перелистнуть конфликты | Показать следующие 25 карточек без перегрузки интерфейса | Нет |
| Выбрать skip/replace/merge | Явно решить неоднозначность одного телефона | Нет |
| Импортировать контакты | Единственная осознанная точка записи текущего импорта | Да, атомарно |
| Открыть вкладку контактов | Получить одну страницу из 25 записей справочника | Только SELECT |
| Перейти на следующую страницу | Лениво запросить следующие 25 контактов | Только SELECT |
| Найти контакт | Отфильтровать справочник по подстроке без загрузки всей БД | Только SELECT |
| Начать/отменить редактирование | Проверить правки до отправки | Нет |
| Сохранить контакт | Оперативно исправить актуальные поля одной записи | Да, одна транзакция |
| Предпросмотр рассылки | Проверить сообщения по всему справочнику | Только SELECT |
| Экспорт рассылки | Получить CSV по всем контактам | Только SELECT |

## Что находится в БД

До commit количество строк в `uploaded_files`, `file_rows`, `contacts` и
`contact_sources` не меняется. После commit:

- `uploaded_files` хранит метаданные подтверждённого импорта, но не бинарный файл;
- `file_rows` хранит полные нормализованные строки этого импорта;
- `contacts` хранит текущее состояние уникальных контактов по телефону;
- `contact_sources` связывает контакт с подтверждённой строкой и решением.

Рассылка всегда вызывает `storage.ListContacts` и работает по всем контактам БД.
Локальный черновик и конкретный последний файл на состав рассылки не влияют.

Вкладка «Контакты» использует `storage.ListContactsPage`. В нём видны два прямых
SQL-запроса: `SELECT count(*)` для количества совпадений и
`SELECT ... ORDER BY id LIMIT $2 OFFSET $3` для текущей страницы. Параметр `q`
ищется как регистронезависимая подстрока по UID, телефону, email, ФИО и скидке.
Размер страницы зафиксирован на backend и равен 25.

Ручное сохранение вызывает `PUT /api/contacts/{uid}`. Сервис нормализует телефон,
email и скидку, после чего `storage.UpdateContact` открывает serializable-
транзакцию. Старый и новый телефоны блокируются advisory lock в сортированном
порядке, строка читается через `SELECT ... FOR UPDATE`, а версия сравнивается с
переданным `updatedAt`. Только после этого выполняется прямой `UPDATE contacts`.
`uid`, `created_at` и связи `contact_sources` пользователь не меняет.

## Ошибочные и конкурентные сценарии

- В черновике есть ошибки: preview и commit возвращают `422`, запись запрещена.
- Пользователь закрыл страницу: черновик восстанавливается из IndexedDB.
- Пользователь отменил импорт: удаляется только локальный черновик.
- Один importId отправлен повторно: commit возвращает `409`.
- Контакт изменился после preview: версия не совпадает, возвращается `409`, старые
  решения удаляются и интерфейс требует новый preview.
- Контакт изменился после начала ручного редактирования: `PUT` возвращает `409`,
  введённые поля остаются в форме до решения пользователя.
- Новый телефон уже занят другим контактом: уникальное ограничение возвращает
  контролируемый `409`, существующие записи не изменяются.
- Пользователь уходит со страницы или запускает поиск во время редактирования:
  интерфейс просит подтвердить отмену несохранённых изменений.
- SQL упал посередине: транзакция откатывается целиком.
- Два импорта одновременно содержат один телефон: advisory lock сериализует их,
  второй импорт должен перепроверить изменившиеся данные.

## Отложенный этап

Проверка полного обязательного набора колонок таблицы в этот этап намеренно не
входит. Сейчас сохраняется существующее требование телефонной колонки и текущие
правила полей. Контракт обязательных колонок должен быть согласован отдельно,
после чего таблица целиком или отдельные строки смогут получать специальный
статус «не соответствует формату».
