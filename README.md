# Загрузка клиентских данных

Go-приложение для локальной подготовки CSV/XLS/XLSX, проверки строк,
подтверждённого импорта контактов и генерации уведомлений по шаблону.

PostgreSQL является обязательным хранилищем. In-memory режима нет.

## Структура проекта

```text
.
├── main.go                         # запуск HTTP-сервера и режим migrate
├── swagger.go                      # Swagger UI и настройка его доступности
├── docs/                           # сгенерированные Swagger 2.0-файлы
├── frontend/                       # встроенный frontend без этапа сборки
│   ├── index.html                  # только HTML-разметка
│   ├── css/                        # базовые, компонентные и feature-стили
│   └── js/                         # ES-модули, IndexedDB, UI-сценарии
├── handlers/                       # HTTP-обработчики
├── models/                         # общие структуры данных
├── services/                       # парсинг и бизнес-логика
├── storage/
│   ├── store.go                    # интерфейсы хранилища
│   ├── postgres_storage.go         # соединение и пул PostgreSQL
│   ├── postgres_imports.go         # атомарная транзакция импорта
│   ├── postgres_file_rows.go       # подготовка подтверждённых строк
│   ├── postgres_contacts.go        # поиск и безопасное изменение контактов
│   ├── migrations.go               # запуск версионированных миграций
│   └── migrations/                 # парные SQL up/down-миграции
├── utils/                          # чистые вспомогательные функции
├── Dockerfile
└── compose.yml
```

SQL-запросы CRUD объявляются локальными константами в начале соответствующих
методов repository. При чтении функции сразу виден полный запрос и его параметры.
DDL не хранится строкой в Go: схема вынесена в SQL-миграции.

Frontend не требует npm или отдельного этапа сборки. `frontend/index.html`
содержит каркас, представления лежат в `frontend/views`, CSS разделён по
назначению, а JavaScript использует нативные ES-модули. Локальный черновик хранит
`frontend/js/draft-store.js` в IndexedDB браузера. `import-workflow.js` управляет
импортом на первой вкладке, а `contacts.js` независимо загружает справочник БД.

## Запуск через Docker Compose

Создайте локальный `.env`:

```bash
cp .env.example .env
```

Обязательно замените `POSTGRES_PASSWORD`.

```bash
docker compose up --build
```

Приложение откроется по адресу:

```text
http://localhost:8080
```

Интерактивная API-документация:

```text
http://localhost:8080/swagger
```

Если порт занят:

```bash
HOST_PORT=18082 docker compose up --build
```

Compose запускает три сервиса:

- `db` — PostgreSQL 16 с постоянным volume;
- `migrate` — применяет ещё не выполненные SQL-миграции и завершается;
- `app` — запускается только после успешных миграций и healthcheck БД.

Остановка без удаления данных:

```bash
docker compose down
```

Удаление контейнеров вместе с локальными данными PostgreSQL:

```bash
docker compose down -v
```

Последнюю команду нельзя использовать для production-базы.

## Запуск без Compose

Сначала применяются миграции:

```bash
DATABASE_URL='postgres://user:password@localhost:5432/task1?sslmode=disable' \
  go run . migrate
```

После этого запускается сервер:

```bash
DATABASE_URL='postgres://user:password@localhost:5432/task1?sslmode=disable' \
  go run .
```

Без `DATABASE_URL` приложение завершится с ошибкой.

Откат последней миграции:

```bash
DATABASE_URL='postgres://user:password@localhost:5432/task1?sslmode=disable' \
  go run . migrate down
```

Команда `down` может удалить таблицы и данные. Перед её запуском приложение нужно
остановить, а для рабочей базы подготовить проверенную резервную копию.

## Модель хранения

### `uploaded_files`

Метаданные только подтверждённого импорта: имя, формат, кодировка, лист Excel,
заголовки, статистика и предупреждения. При выборе и первичной проверке файла
записи здесь ещё нет. Сам бинарный CSV/XLS/XLSX в PostgreSQL не сохраняется.

### `file_rows`

После финального подтверждения каждая нормализованная строка сохраняется вместе
с исходным номером. Невалидный черновик до этой транзакции не допускается.

### `contacts`

Актуальное состояние контакта с фиксированными полями `phone`, `email`, `name`
и `discount`. Внутренний `id` создаётся PostgreSQL как `SERIAL` и используется
для связей между таблицами. Публичный `uid` имеет тип UUID, генерируется через
`gen_random_uuid()` и не зависит от последовательности внутренних ID. Номер
телефона уникален и является ключом дедупликации. Контакт не принадлежит одному
конкретному файлу.

Произвольного поля `data` в контакте нет. Все колонки подтверждённой строки
сохраняются в `file_rows.values`, но в `contacts` попадают только фиксированные
поля справочника.

### `contact_sources`

Связывает числовой `contact_id` с файлом и исходной строкой. Здесь фиксируется
результат обработки строки: `created`, `matched`, `skipped`, `replaced` или
`merged`. Повторное решение обновляет существующую связь, поэтому
история входящих значений и старых состояний контакта не накапливается.

## Поток данных

1. Пользователь выбирает файл. `POST /api/upload` временно читает запрос, парсит и
   нормализует таблицу, создаёт `importId` и возвращает полный черновик. SQL на
   этом этапе не выполняется.
2. Браузер хранит черновик в IndexedDB до 24 часов. Поиск и выбор решений
   конфликтов выполняются локально. Конфликты отображаются по 25 карточек, а
   решения сохраняются при переключении страниц. Правки автоматически остаются
   в IndexedDB.
3. `POST /api/imports/validate` повторно нормализует отредактированный черновик.
   Этот endpoint не обращается к PostgreSQL.
4. `POST /api/imports/preview` выполняет только `SELECT` контактов по телефонам и
   возвращает количества новых, совпадающих и конфликтующих записей.
5. Пользователь выбирает `skip`, `replace` или `merge`. Решения остаются в
   браузере и сами по себе не изменяют БД.
6. Только `POST /api/imports/commit` повторяет проверку и одной serializable-
   транзакцией создаёт `uploaded_files`, копирует `file_rows`, изменяет `contacts`
   и добавляет `contact_sources`. Любая ошибка откатывает всю транзакцию.
7. После успешного commit локальный черновик удаляется. При отмене он удаляется
   без SQL-запросов.
8. Вкладка «Контакты» вызывает `GET /api/contacts?page=N&q=...`. Backend ищет
   подстроку по телефону, ФИО, email, скидке и UID, выполняет `COUNT(*)` и
   загружает только 25 совпадений через `LIMIT/OFFSET`.
9. Ручное сохранение отправляет `PUT /api/contacts/{uid}`. Backend повторно
   нормализует поля и обновляет строку только при совпадении версии `updatedAt`.
   Уникальность телефона и advisory lock защищают от конфликтов с импортом.
10. Предпросмотр и экспорт рассылки читают все актуальные строки `contacts`, а не
   текущий черновик и не один выбранный файл.

Подробное обоснование экранов, функций и сценариев находится в
[`docs/IMPORT_WORKFLOW.md`](docs/IMPORT_WORKFLOW.md).

## Миграции

Миграции находятся в `storage/migrations`, хранятся парами `up/down` и встраиваются
в бинарник через `go:embed`. Начальная миграция сразу создаёт актуальную схему без
legacy-колонок и переходных типов. Следующие изменения схемы добавляются новыми
пронумерованными парами файлов.

Перед выполнением используется PostgreSQL advisory lock: два одновременно
запущенных мигратора не смогут изменять схему параллельно. Каждая серия миграций
выполняется транзакционно.

Для каждой версии сохраняются имя и SHA-256 SQL-файла. При старте приложение
сравнивает полный список миграций, поэтому пропущенная версия или изменение уже
применённого SQL останавливает запуск. Применённые миграции не редактируются:
любое последующее изменение оформляется новым номером.

`server migrate down` откатывает ровно последнюю применённую версию в транзакции
и удаляет её запись из `schema_migrations`. В production предпочтительнее новая
исправляющая миграция: down используется только для контролируемого отката и не
заменяет резервную копию данных.

Для production рекомендуется запускать `server migrate` отдельной учётной
записью с DDL-правами. HTTP-приложению достаточно прав на чтение и изменение
рабочих таблиц.

## Основные переменные окружения

| Переменная | По умолчанию | Назначение |
| --- | --- | --- |
| `DATABASE_URL` | нет | Обязательная строка подключения PostgreSQL |
| `HOST_PORT` | `8080` | Порт приложения на хосте в Compose |
| `PORT` | `8080` | Порт HTTP-сервера внутри контейнера |
| `MAX_UPLOAD_SIZE_MB` | `20` | Максимальный размер файла |
| `MAX_UPLOAD_SIZE_BYTES` | нет | Точный лимит, имеет приоритет над MB |
| `SWAGGER_ENABLED` | `true` | Включает Swagger UI и JSON-спецификацию |
| `DB_MAX_CONNS` | `10` | Максимум соединений пула |
| `DB_MIN_CONNS` | `0` | Минимум соединений пула |
| `DB_CONNECT_TIMEOUT_SECONDS` | `10` | Таймаут подключения |
| `DB_QUERY_TIMEOUT_SECONDS` | `5` | Таймаут CRUD-запросов |
| `DB_IMPORT_TIMEOUT_SECONDS` | `45` | Таймаут preview и атомарного импорта |
| `DB_MIGRATION_TIMEOUT_SECONDS` | `60` | Таймаут миграций |
| `DB_MAX_CONN_IDLE_TIME_SECONDS` | `1800` | Максимальный простой соединения |
| `DB_MAX_CONN_LIFETIME_SECONDS` | `3600` | Максимальная жизнь соединения |
| `DB_HEALTH_CHECK_SECONDS` | `30` | Период проверки соединений пула |

## API

Swagger UI содержит тела запросов, примеры, допустимые значения и все коды
ответов. Короткий адрес `/swagger` перенаправляет на интерфейс, а исходная
Swagger 2.0-спецификация доступна по `/swagger/doc.json`.

| Метод | Маршрут | Назначение |
| --- | --- | --- |
| `GET` | `/api/health` | Проверка приложения и PostgreSQL |
| `POST` | `/api/upload` | Stateless-разбор файла без записи в БД |
| `POST` | `/api/imports/validate` | Повторная проверка локальных правок |
| `POST` | `/api/imports/preview` | Read-only сравнение с контактами БД |
| `POST` | `/api/imports/commit` | Атомарный подтверждённый импорт |
| `GET` | `/api/contacts?page=N&q=...` | Поиск и страница из 25 подтверждённых контактов |
| `PUT` | `/api/contacts/{uid}` | Сохранение проверенных полей одного контакта |
| `POST` | `/api/preview` | Предпросмотр уведомлений |
| `POST` | `/api/export` | Экспорт уведомлений в CSV |

Аннотации находятся рядом с соответствующими обработчиками в `handlers`.
После изменения API документация пересобирается закреплённой версией Swag:

```bash
go generate ./...
```

Если Go установлен только через Docker:

```bash
docker run --rm \
  -v "$PWD":/src \
  -w /src \
  golang:1.22-alpine \
  go generate ./...
```

Команда обновляет `docs/docs.go`, `docs/swagger.json` и `docs/swagger.yaml`.
Эти файлы входят в репозиторий и Docker-образ, поэтому для запуска приложения
CLI `swag` не нужен.

## Проверка

Все Go-тесты:

```bash
go test ./...
```

Тесты `services/import_service_test.go` проверяют нормализацию, запрет импорта
невалидного черновика, классификацию контактов и версии конфликтов.
`handlers/import_test.go` отдельно доказывает, что preview не вызывает commit.

## Production

- Используйте длинный уникальный пароль без значения по умолчанию.
- Для внешней PostgreSQL включайте TLS (`sslmode=verify-full`).
- Не публикуйте порт PostgreSQL в интернет.
- Разделяйте роли мигратора и HTTP-приложения.
- Настройте автоматические резервные копии и регулярно проверяйте восстановление.
- Перед обновлением сначала применяйте миграции, затем запускайте новую версию app.
- Отключайте Swagger через `SWAGGER_ENABLED=false`, если документация не должна
  быть доступна из production-сети.
