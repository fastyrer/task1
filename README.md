# Загрузка клиентских данных

Go-приложение для загрузки CSV/XLS/XLSX, проверки строк, поиска по содержимому,
сохранения контактов и генерации уведомлений по шаблону.

PostgreSQL является обязательным хранилищем. In-memory режима нет.

## Структура проекта

```text
.
├── main.go                         # запуск HTTP-сервера и режим migrate
├── index.html                      # встроенный frontend
├── handlers/                       # HTTP-обработчики
├── models/                         # общие структуры данных
├── services/                       # парсинг и бизнес-логика
├── storage/
│   ├── store.go                    # интерфейсы хранилища
│   ├── postgres_storage.go         # соединение и пул PostgreSQL
│   ├── postgres_files.go           # файлы, строки и поиск
│   ├── postgres_contacts.go        # контакты, конфликты и аудит
│   ├── migrations.go               # запуск версионированных миграций
│   └── migrations/                 # парные SQL up/down-миграции
├── utils/                          # чистые вспомогательные функции
├── Dockerfile
└── compose.yml
```

SQL-запросы CRUD объявляются локальными константами в начале соответствующих
методов repository. При чтении функции сразу виден полный запрос и его параметры.
DDL не хранится строкой в Go: схема вынесена в SQL-миграции.

## Запуск через Docker Compose

Создайте локальный `.env`:

```bash
cp .env.example .env
```

Обязательно замените `POSTGRES_PASSWORD`.

```bash
docker compose up --build
```

Приложение откроется по адресу:

```text
http://localhost:8080
```

Если порт занят:

```bash
HOST_PORT=18082 docker compose up --build
```

Compose запускает три сервиса:

- `db` — PostgreSQL 16 с постоянным volume;
- `migrate` — применяет ещё не выполненные SQL-миграции и завершается;
- `app` — запускается только после успешных миграций и healthcheck БД.

Остановка без удаления данных:

```bash
docker compose down
```

Удаление контейнеров вместе с локальными данными PostgreSQL:

```bash
docker compose down -v
```

Последнюю команду нельзя использовать для production-базы.

## Запуск без Compose

Сначала применяются миграции:

```bash
DATABASE_URL='postgres://user:password@localhost:5432/task1?sslmode=disable' \
  go run . migrate
```

После этого запускается сервер:

```bash
DATABASE_URL='postgres://user:password@localhost:5432/task1?sslmode=disable' \
  go run .
```

Без `DATABASE_URL` приложение завершится с ошибкой.

Откат последней миграции:

```bash
DATABASE_URL='postgres://user:password@localhost:5432/task1?sslmode=disable' \
  go run . migrate down
```

Команда `down` может удалить таблицы и данные. Перед её запуском приложение нужно
остановить, а для рабочей базы подготовить проверенную резервную копию.

## Модель хранения

### `uploaded_files`

Метаданные загрузки: имя, формат, кодировка, лист Excel, заголовки, статистика и
предупреждения. Строки файла в этой таблице не дублируются.

### `file_rows`

Каждая строка файла хранится отдельно вместе с исходным номером строки,
признаком валидности, ошибками и подготовленным `search_text`. Поиск выполняется
PostgreSQL с trigram-индексом, поэтому сервер не загружает весь файл для поиска.

### `contacts`

Актуальное состояние контакта с фиксированными полями `phone`, `email`, `name`
и `discount`. Внутренний `id` создаётся PostgreSQL как `SERIAL` и используется
для связей между таблицами. Публичный `uid` имеет тип UUID, генерируется через
`gen_random_uuid()` и не зависит от последовательности внутренних ID. Номер
телефона уникален и является ключом дедупликации. Контакт не принадлежит одному
конкретному файлу.

Произвольного поля `data` в контакте нет. Все исходные колонки загруженной строки
по-прежнему сохраняются в `file_rows.values`, но в таблицу контактов не копируются.

### `contact_sources`

Связывает числовой `contact_id` с файлом и исходной строкой. Здесь фиксируется
текущее решение для этой строки: `created`, `matched`, `skipped`, `replaced`,
`merged` или `fixed`. Повторное решение обновляет существующую связь, поэтому
история входящих значений и старых состояний контакта не накапливается.

## Поток данных

1. `POST /api/upload` читает CSV/XLS/XLSX, нормализует строки и вызывает
   `SaveFileData`. Метаданные попадают в `uploaded_files`, все строки файла — в
   `file_rows`. После ответа обработчика данные уже находятся в PostgreSQL.
2. Поиск получает `fileId` и выполняет `ILIKE` непосредственно по
   `file_rows.search_text`.
3. Предпросмотр и экспорт рассылки получают только шаблон, выполняют прямой
   `SELECT` актуальных записей из `contacts` и формируют по одному сообщению на
   уникальный телефон. Загруженный в данный момент файл на состав рассылки не влияет.
4. `POST /api/contacts/save` отдельно переносит валидные строки выбранного файла в
   общий справочник `contacts`. После разрешения конфликтов именно текущее состояние
   этого справочника используется в рассылке.
5. Каждый загруженный файл имеет собственный UUID. Фронтенд работает с последним
   загруженным `fileId` только в операциях поиска, сохранения и исправления строк.
   Рассылка автоматически охватывает контакты, сохранённые из всех файлов.

## Миграции

Миграции находятся в `storage/migrations`, хранятся парами `up/down` и встраиваются
в бинарник через `go:embed`. Начальная миграция сразу создаёт актуальную схему без
legacy-колонок и переходных типов. Следующие изменения схемы добавляются новыми
пронумерованными парами файлов.

Перед выполнением используется PostgreSQL advisory lock: два одновременно
запущенных мигратора не смогут изменять схему параллельно. Каждая серия миграций
выполняется транзакционно.

Для каждой версии сохраняются имя и SHA-256 SQL-файла. При старте приложение
сравнивает полный список миграций, поэтому пропущенная версия или изменение уже
применённого SQL останавливает запуск. Применённые миграции не редактируются:
любое последующее изменение оформляется новым номером.

`server migrate down` откатывает ровно последнюю применённую версию в транзакции
и удаляет её запись из `schema_migrations`. В production предпочтительнее новая
исправляющая миграция: down используется только для контролируемого отката и не
заменяет резервную копию данных.

Для production рекомендуется запускать `server migrate` отдельной учётной
записью с DDL-правами. HTTP-приложению достаточно прав на чтение и изменение
рабочих таблиц.

## Основные переменные окружения

| Переменная | По умолчанию | Назначение |
| --- | --- | --- |
| `DATABASE_URL` | нет | Обязательная строка подключения PostgreSQL |
| `HOST_PORT` | `8080` | Порт приложения на хосте в Compose |
| `PORT` | `8080` | Порт HTTP-сервера внутри контейнера |
| `MAX_UPLOAD_SIZE_MB` | `20` | Максимальный размер файла |
| `MAX_UPLOAD_SIZE_BYTES` | нет | Точный лимит, имеет приоритет над MB |
| `SEARCH_RESULT_LIMIT` | `1000` | Максимум результатов поиска |
| `DB_MAX_CONNS` | `10` | Максимум соединений пула |
| `DB_MIN_CONNS` | `0` | Минимум соединений пула |
| `DB_CONNECT_TIMEOUT_SECONDS` | `10` | Таймаут подключения |
| `DB_QUERY_TIMEOUT_SECONDS` | `5` | Таймаут CRUD-запросов |
| `DB_MIGRATION_TIMEOUT_SECONDS` | `60` | Таймаут миграций |
| `DB_MAX_CONN_IDLE_TIME_SECONDS` | `1800` | Максимальный простой соединения |
| `DB_MAX_CONN_LIFETIME_SECONDS` | `3600` | Максимальная жизнь соединения |
| `DB_HEALTH_CHECK_SECONDS` | `30` | Период проверки соединений пула |

| --- | --- | --- |
| `GET` | `/api/health` | Проверка приложения и PostgreSQL |

## Проверка

Все Go-тесты:

```bash
go test ./...
```

Интеграционный тест PostgreSQL запускается только при наличии отдельной тестовой
БД. Её имя обязательно должно содержать `test`:

```bash
TEST_DATABASE_URL='postgres://user:password@localhost:5432/task1_test?sslmode=disable' \
  go test ./storage -run TestCleanPostgresSchemaAndContactResolution -v
```

Тест проверяет цикл `up → down → up`, чистую схему, генерацию внутреннего
`SERIAL id` и публичного `uid`, разрешение конфликтов без накопления истории и
атомарное исправление строки.

## Production

- Используйте длинный уникальный пароль без значения по умолчанию.
- Для внешней PostgreSQL включайте TLS (`sslmode=verify-full`).
- Не публикуйте порт PostgreSQL в интернет.
- Разделяйте роли мигратора и HTTP-приложения.
- Настройте автоматические резервные копии и регулярно проверяйте восстановление.
- Перед обновлением сначала применяйте миграции, затем запускайте новую версию app.
